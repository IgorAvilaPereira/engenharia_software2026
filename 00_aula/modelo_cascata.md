## Modelo em cascata

O **modelo em cascata** (também chamado de **Modelo em Cascata** ou *Waterfall Model*) se encaixa dentro da área de **Engenharia de Software**, mais especificamente nos **modelos de processo de desenvolvimento de software**.

### 📌 Onde ele se encaixa

Ele é um **modelo de processo** usado para organizar as etapas de criação de um sistema. Normalmente aparece quando se estuda:

* **Engenharia de Software**
* **Desenvolvimento de Software**
* **Modelos de Processo de Software**

### 🧩 Como funciona

No modelo em cascata, o desenvolvimento ocorre em **etapas sequenciais**, como uma “cascata”, onde uma fase termina antes da próxima começar:

1. **Análise do sistema e definição de requisitos**
2. **Projeto de sistema e software**
3. **Implementação e teste de unidade**
4. **Integraçaõ e Testes**
5. **Operação e Manutenção**

Cada fase depende da conclusão da anterior.

### 👍 Quando ele é mais usado

Ele funciona melhor quando:

* os **requisitos são bem definidos desde o início**
* o projeto **não terá muitas mudanças**
* há **documentação detalhada**

### ⚠️ Limitações

* Pouca flexibilidade para mudanças.
* Problemas podem aparecer **só no final**, durante os testes.

No **Documento de Especificação de Requisitos de Software (SRS)**, o **Modelo em Cascata** aparece **logo no início do projeto**, na fase de **levantamento e especificação de requisitos**.

### 📄 Onde ele entra no processo

No modelo em cascata, o **SRS é produzido na primeira grande fase**. A sequência costuma ser:

1️⃣ **Levantamento de requisitos**

* Entender o que o cliente precisa.

2️⃣ **Especificação de requisitos**

* Aqui é criado o **Documento de Especificação de Requisitos de Software (SRS)**.
* Ele descreve **tudo que o sistema deve fazer**.

3️⃣ **Validação dos requisitos**

* O cliente revisa o documento.

4️⃣ Depois disso o projeto segue para **design, implementação e testes**.

### 📌 Por que o SRS é tão importante no cascata

No **modelo em cascata**, quase **todas as fases dependem desse documento**. Ele serve como base para:

* **Projeto do sistema (design)**
* **Implementação**
* **Testes**
* **Aceitação do cliente**

Ou seja, no cascata o **SRS funciona como o “contrato técnico” do sistema**.

✅ Resumindo:

* **Modelo em cascata** → define **como o projeto avança em fases**.
* **SRS** → é o **documento produzido na fase de requisitos** dentro desse modelo.
