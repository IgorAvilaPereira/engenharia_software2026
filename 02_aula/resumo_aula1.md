A **elicitação de requisitos** normalmente é registrada dentro do **Documento de Especificação de Requisitos de Software (ERS)**.

Esse documento também é conhecido como **SRS – Software Requirements Specification**, definido no padrão da IEEE.

### Onde aparece dentro do documento

A elicitação não costuma ser uma seção principal isolada; ela aparece como **parte da descrição dos requisitos** ou da **metodologia utilizada** para levantar os requisitos.

Uma estrutura comum do **ERS** é:

1. **Introdução**
2. **Descrição geral do sistema**
3. **Metodologia ou processo de levantamento de requisitos**
   (aqui pode ser descrita a **elicitação**: entrevistas, questionários, observação etc.)
4. **Requisitos funcionais**
5. **Requisitos não funcionais**
6. **Regras de negócio**
7. **Casos de uso**
8. **Prioridade dos requisitos** (MoSCoW, Kano etc.)

### Resumindo

* **Elicitação** → processo de levantar necessidades dos usuários
* **ERS / SRS** → documento onde os requisitos levantados são **registrados e organizados**

✅ Ou seja, a **elicitação gera as informações que são documentadas na ERS**.

---

### Elicitação de requisitos

É a etapa em que se **descobre e coleta as necessidades do usuário e do sistema**.

Exemplos de técnicas:

* Entrevistas
* Questionários
* Observação
* Workshops
* Brainstorming

Resultado: **lista inicial de requisitos**.

---

### Análise e priorização de requisitos

Depois de coletados, os requisitos são **avaliados e organizados por importância**.

Métodos comuns:

* **MoSCoW** (Must, Should, Could, Won't)
* **Kano**
* Valor de negócio vs. esforço
* Análise de risco

Resultado: **requisitos priorizados**.

---

### Especificação de requisitos

Aqui os requisitos são **documentados formalmente**, normalmente em um **ERS (Especificação de Requisitos de Software)**.

Tipos principais:

**Requisitos Funcionais**

* Descrevem **o que o sistema deve fazer**.

Exemplo:

* O sistema deve permitir cadastro de usuários.
* O sistema deve gerar relatório de vendas.

**Requisitos Não Funcionais**

* Descrevem **características de qualidade do sistema**.

Exemplo:

* O sistema deve responder em até 2 segundos.
* O sistema deve suportar 500 usuários simultâneos.

---

### Fluxo recomendado (simplificado)


1. Introdução

2. Descrição geral do sistema

3. Metodologia de elicitação de requisitos

* entrevistas

* questionários

* prototipação

4. Requisitos funcionais

5. Requisitos não funcionais

6. Casos de uso 

7. Priorização de requisitos

8. Protótipos de interface (opcional)
